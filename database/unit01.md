## 데이터 모델링 기본

**데이터 모델링**은 현실 세계의 데이터를 데이터베이스로 옮기기 위해 구조화하고 약속된 표기법으로 시각화하는 과정이다. **개념적**, **논리적**, **물리적** 모델링까지 총 3단계로 진행된다.

<br>

### 1. 데이터 모델링의 3단계

**개념적 모델링(Conceptual Modeling)**

- **정보 모델링**이라고도 하며, 요구사항 분석을 기반으로 전체적인 뼈대를 만드는 과정
- 엔터티를 추출하고 엔터티 간의 관계를 정의하여 **E-R 다이어그램**을 작성

  <img src="https://raw.githubusercontent.com/Team-Gravit/gravit-images/main/database/unit01/image1.png" width="100%">

<br>
<br>

**논리적 모델링(Logical Modeling)**

- 개념적 모델링에서 만든 E-R 다이어그램을 DBMS에 맞게 매핑하는 과정
- 상세 속성을 추출하고 정규화를 수행

<br>

**물리적 모델링(Physical Modeling)**

- 논리적 모델을 실제 저장하기 위해 물리적 구조를 정의하는 과정
- DBMS 특성에 맞는 저장 구조를 정의해야 최적의 성능을 얻음

<br>

### 2. 엔터티(Entity)

**엔터티**는 업무에서 필요하고 유용한 정보를 저장하고 관리하기 위한 집합적인 것으로, 물리적 데이터베이스에서는 **테이블**(Table)에 해당한다.

<br>

**엔터티의 특징**

- 유일한 식별자로 구분할 수 있어야 함
- 반드시 속성을 가져야 하며, 다른 엔터티와 최소 1개 이상의 관계를 가짐

<br>

### 3. 속성(Attribute)

**속성**은 업무에서 필요한 인스턴스의 구성요소로, 의미상 더 이상 분리되지 않는 최소 데이터 단위이다. 물리적 데이터베이스에서는 **컬럼**(Column)에 해당한다.

<br>

**특성에 따른 분류**

| **유형** | **설명** |
| --- | --- |
| 기본 속성 | 업무에서 추출한 원래 속성 |
| 설계 속성 | 모델링 과정에서 새로 만든 속성 |
| 파생 속성 | 다른 속성을 통해 계산된 속성 |

<br>

**엔터티 구성에 따른 분류**

| **유형** | **설명** |
| --- | --- |
| PK 속성 | 엔터티를 유일하게 식별하는 속성 |
| FK 속성 | 다른 엔터티와의 관계를 나타내는 속성 |
| 일반 속성 | PK, FK가 아닌 나머지 속성 |

<br>

### 4. 인스턴스(Instance)

**인스턴스**는 엔터티에 저장된 개별 데이터를 의미한다. 물리적 데이터베이스에서는 **행**(Row) 또는 **튜플**(Tuple)에 해당한다.

<br>
<br>

### 5. 관계(Relationship)

관계는 **엔터티 간의 연관성**을 나타낸다. 두 엔터티가 업무적으로 어떻게 연결되어 있는지 정의한다.

**관계의 종류**

| **유형** | **설명** | **예시** |
| --- | --- | --- |
| 일대일(1:1) | 양쪽 엔터티가 각각 하나씩 대응 | 사원 - 사원 정보 |
| 일대다(1:N) | 한쪽 엔터티가 여러 개와 대응 | 부서 - 사원 |
| 다대다(M:N) | 양쪽 모두 여러 개와 대응 | 학생 - 강좌 |

<br>

> **💡다대다**(M:N) 관계는 물리적 데이터베이스에서 직접 구현할 수 없으므로, **연결 엔터티**(중간 테이블)을 추가하여 **일대다**(1:N) 관계로 해소한다.

<br>


### 6. 카디널리티(Cardinality)

카디널리티는 두 가지 의미로 사용된다.

<br>

**관계에서의 카디널리티**

- 두 테이블 간의 **관계에서 각각의 레코드가 서로 어떻게 연결**되는지를 나타냄
- 1:1, 1:N, M:N 등으로 표현

<br>

**컬럼에서의 카디널리티**

- 특정 컬럼에 존재하는 **고유한 값의 개수**를 의미
- 인덱스 설계 시 중요한 판단 기준
- 주민등록번호 → 카디널리티 높음, 성별 → 카디널리티 낮음
